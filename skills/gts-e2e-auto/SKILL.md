# E2E 自动测试（硬性操作规程）

> 触发词：兄弟说「e2e自动」/「自动e2e」/「自动测试」，可选附带操作描述和验证目标。
> 动态生成测试脚本 → 采集自定义数据 → 验证目标 → 结果分析 + 飞书短通知。

---

## 步骤

### Step 1：解析输入

#### 情况 A：兄弟明确指定了操作和验收

兄弟格式示例：
```
e2e自动测试：
操作巨大娘往小人方向移动，使其在移动时与小人碰撞；
验证：小人血量至少减少5点
```

**解析出：**
- **操作序列：** 巨大娘向小人方向移动，制造碰撞
- **验证目标：** 小人血量至少减少 5 点

**设计专用日志：** 在注入页面的 logger 中增加自定义 hook，采集验证所需的数据字段。

→ 直接跳到 Step 2。

#### 情况 B：兄弟没有指定操作和/或验收

如果兄弟只说了「e2e自动」但没有指定操作序列或验证目标：

1. **分析最近改动：** 检查 `git log --oneline -5` + `git diff HEAD~1`（或指定 SHA），理解最近修改了什么
2. **生成方案：** 基于改动内容，设计：
   - **操作序列：** 模拟什么用户操作能触发改动的代码路径
   - **验证目标：** 具体、可量化的验收标准（数值变化、状态切换、错误消失等）
3. **输出方案待确认：** 在会话中输出方案，等兄弟明确说「开搞」「执行」「上」「可以」等确认词
4. **同时发飞书通知：** 附带方案摘要，提醒兄弟确认
5. 兄弟确认后，进入 Step 2

### Step 2：设计日志 + 生成脚本

**设计专用日志：** 在注入页面的 logger 中增加自定义 hook，采集验证所需的数据字段。

以 `e2e-scenario2-auto.cjs` 为模板，修改：
- **操作部分：** 替换为方案中确定的操作序列
- **日志采集：** 注入 Step 1/2 设计的专用 hook
- **验收逻辑：** 替换为指定的验证条件

脚本内置：
- 双 tab 自动登录（P1 创房，P2 加入）
- 执行操作序列
- 专用数据采集
- 最多重试 **5 次**，总超时 **10 分钟**
- **重试前必须：关闭上一轮的所有 Chromium/Playwright 进程 + 等待 2 秒释放端口**

生成后保存到 `test/e2e/auto/`，文件名包含日期和描述，不删除。

以 background 模式运行，启动后告知兄弟。

### Step 3：验证 + 会话分析

脚本退出后，检查采集到的数据，在会话中输出分析结果，然后发飞书短通知（≤10 字）。

### 超限处理（超过最大尝试次数 5 次或总超时 10 分钟）

如果超过重试限制后仍未通过，脚本必须：
1. **停止重试**，不再继续
2. 在会话中输出**完整分析报告**，包括：
   - 每次尝试的失败原因
   - 失败模式分布（同一原因反复 vs 多种不同原因）
   - 建议下一步排查方向
3. 发飞书通知：`❌ E2E自动测试超限失败`（附原因摘要）

---

## 执行纪律

1. **每次动态生成脚本**，不跑固定脚本
2. **脚本保留不删除**——保存到 `test/e2e/auto/`
3. 日志设计要**精准**——只采集验证目标需要的字段
4. 重试上限 **5 次**，总超时 **10 分钟**。**每次重试前必须先 kill 上一轮的 Playwright/Chromium 进程**
5. 超限后**给出分析报告 + 飞书通知**，不能静默退出
6. 结果分析**在会话中给出**，飞书只发短提醒（≤10 字）
7. 和 `gts-e2e-test`（手动双窗口版）互不触发
