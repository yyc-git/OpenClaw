# 代码审核（硬性操作规程）

> 触发词：兄弟说「代码审核」或「审核」。
> 全手动交互模式：每步等兄弟确认后才继续。
> 🔴 = 方案中列出确认后改 | 🟡 = 出方案等确认后改 | 🟢 = 只记不改

---

## 步骤

### Step 1：扫描改动

- 读 `.last-review`（`笔记/决策记录/`），获取上次审核的 commit
- `git diff <last-review>..HEAD` 出改动文件清单
- 筛掉：`.gitignore` 中匹配的内容、`node_modules/`、`dist/`、`*.gen.tsx`
- ⚠️ **不筛 `test/`**，测试代码也在审核范围内

输出格式：
```
## 代码审核 — Step 1：改动扫描

**上次审核：** <commit>
**改动文件：** <N> 个（列出）
**改动规模：** +<N> / -<N> 行
**请兄弟确认后进入 Step 2：逐文件审查**
```

### Step 2：逐文件审查 + 出重构方案

对每个改动文件执行以下检查，按优先级输出：

#### 🔴 清理项（方案中列出，兄弟确认后一起改）

| 规则 | 说明 |
|---|---|
| 测试代码残留 | `window.__xxx`、`[DBG]`、`__` 前缀的全局变量、调试用 `console.log` |
| 未使用 import | 删除未被引用的 import 语句 |
| 多余的文件/代码 | 无引用的文件、死代码块、被注释掉的旧实现 |

#### 🟡 重构项（出方案，等兄弟确认后才改）

| 规则 | 说明 |
|---|---|
| 文件 >500 行 | 按 export 拆文件 |
| 函数 >100 行 | 提子函数 |
| 重复代码 ≥3 处 | 提公共函数 |
| `any` 类型使用 | 替换为具体类型 |
| 非纯函数混入副作用 | 应纯化的函数却访问了全局状态/模块级变量/DOM |
| 状态同步违规 | 发送变化量而非绝对值；客户端不等 server 确认就改本地状态 |
| 源文件不同步 | 改 `.gen.tsx` 未同步 `.res` / 改 `dist` 未同步 `src` |

#### 🟢 关注项（只记不改）

| 规则 | 说明 |
|---|---|
| 混合 logic+render 职责 | 记摘要 |
| 条件嵌套 >3 层 | 加 `// TODO: 降低复杂度` |
| 导出函数缺 JSDoc | 补占位注释 |
| 空 catch 块静默吞错 | 记摘要 |
| 未清理的 event listener/timer | 记摘要（内存泄漏风险） |

输出格式：
```
## 代码审核 — Step 2：审查结果 + 重构方案

### 🔴 清理项（<N> 处）
- <文件>：<问题>

### 🟡 待确认重构（<N> 处）
- <文件>：<问题> → <方案>

### 🟢 关注项（<N> 处）
- <文件>：<问题>

**请兄弟确认：哪些 🟡 要改？**
```

> ⏸️ **在此停住等确认。** 兄弟确认后，执行已确认的 🔴 + 🟡，完成后进入 Step 3。

### Step 3：验证

- 重构完成后，跑全部 BDD 测试：`npx jest --config jest.multiplayer.json`
- 测试不通过 → 修到全绿
- 测试全绿 → 进入 Step 4

### Step 4：输出审核摘要 + 提交 + 更新文档

输出格式：
```
## 代码审核摘要 — <日期>

**改动文件：** <N> 个
**🔴 已清理：** <N> 处
**🟡 已重构：** <N> 处（列出改动）
**🟢 关注项：** <N> 处
**测试结果：** ✅ 全绿 / ❌ <N> 失败
**重构效果：** <分析>

**请兄弟确认后更新文档**
```

> ⏸️ **在此停住等确认。** 兄弟确认后：
> 1. 更新 `笔记/` 中对应目录的文档（架构级 → `项目文档/`+ADR、重构 → `方案/`+`代码笔记/`、调试 → `讨论记录/`）
> 2. 更新 `.last-review = HEAD`
> 3. `git add -A` + `git commit`（提交git，不 push）
> 4. 发飞书通知总结
> 5. 提示兄弟：「审核完成 ✅ 建议跑 e2e测试 确认联机环境无误，确认后推送」

---

## 执行纪律

1. **纯手动模式**——每步都停住等兄弟确认，不自动往下走
2. 🔴 清理项在 Step 2 方案中列出，兄弟确认后一起改
3. 🟡 重构必须先出方案，兄弟明确说改哪个才改
4. 🟢 只记录不改，不擅自动业务逻辑
5. 重构后测试不过 → 修到过，不能留烂摊子
6. Step 2 确认后 → 改代码 → Step 3 测试 → Step 4 摘要 → 提交，顺序不能乱
7. 审核只审已有代码，不混入新功能
8. **保存协议不触发此 Skill**，代码审核是独立流程
