# 代码审核（硬性操作规程）

> 触发词：兄弟说「代码审核」或「审核」。
> 全手动交互模式：每步等兄弟确认后才继续。
> 🔴 = 方案中列出确认后改 | 🟡 = 出方案等确认后改 | 🟢 = 只记不改
> 📣 每次等待兄弟确认时，必须同时发飞书通知（含审核进度和确认诉求）。

---

## 步骤

### Step 1：扫描改动

- 读 `.last-review`（`笔记/决策记录/`），获取上次审核的 commit
- `git diff <last-review>..HEAD` 出改动文件清单
- 筛掉：`.gitignore` 中匹配的内容、`node_modules/`、`dist/`、`*.gen.tsx`
- ⚠️ **不筛 `test/` 和 `笔记/`**，测试代码和文档也在审核范围内

输出格式：
```
## 代码审核 — Step 1：改动扫描

**上次审核：** <commit>
**改动文件：** <N> 个（列出）
**改动规模：** +<N> / -<N> 行
**请兄弟确认后进入 Step 2：逐文件审查**
```

> ⏸️ **在此停住等确认。** 📣 发飞书通知（审核进度 + 改动文件清单 + 请求确认）。
> 兄弟确认后进入 Step 2。

### Step 2：逐文件审查 + 出重构方案

对每个改动文件执行以下检查，按优先级输出：

#### 🔴 清理项（方案中列出，兄弟确认后一起改）

| 规则 | 说明 |
|---|---|
| 测试代码残留 | `window.__xxx`、`[DBG]`、`__` 前缀的全局变量、调试用 `console.log` |
| 未使用 import | 删除未被引用的 import 语句 |
| 多余的文件/代码 | 无引用的文件、死代码块、被注释掉的旧实现 |

#### 🟡 重构项（出方案，等兄弟确认后才改）

| 规则 | 说明 |
|---|---|
| **架构与结构** | |
| 文件 >500 行 | 按 export 拆文件 |
| 函数 >100 行 | 提子函数 |
| 重复代码 ≥3 处 | 提公共函数 |
| **类型与命名** | |
| `any` 类型使用 | 替换为具体类型 |
| 名字需要重命名 | 变量/函数/类型/枚举名与实际用途不符时标记。场景：① 未清晰表达意图（模糊词/缩写）② 与实现不一致（谎言式命名）③ 业务术语不统一 ④ 过度宽泛 ⑤ 隐喻/模式混淆 ⑥ 框架约定变更。出方案列建议名，确认后改 |
| 测试 mock 过多 → 重构生产函数 | 测试大量 `any` 或巨型 mock → 生产函数依赖太宽，拆小接口/收窄参数 |
| **职责与依赖** | |
| 非纯函数混入副作用 | 应纯化的函数访问了全局状态/模块级变量/DOM |
| 层间职责越界 | 任何层做了不该它做的事（logic/render 混合、客户端算服务端逻辑、business_layer 混渲染）。统一按此条处理，不再细分子规则 |
| 源文件不同步 | 改 `.gen.tsx` 未同步 `.res`、改 `dist` 未同步 `src`、改 `.res` 未同步 `.d.ts` |
| **状态管理** | |
| 禁止 `window` 全局挂载 | 不应通过 `window.__xxx` 暴露内部状态或函数；所有数据放入 state 统一管理 |
| 模块级变量未移入 state | `let` 变量场景重入时手动清空易遗漏 → 放入 state |
| 函数隐式调 readState/writeState | 应通过参数传入 state/mp，顶层编排除外 |
| 函数修改 state/mp 后必须返回 | 修改 state/mp 参数的函数必须返回更新后的值，禁止仅靠引用修改隐式生效 |
| **安全与可靠** | |
| 未使用项目自定义 Three.js class | 直接 `new THREE.Sprite` / `new THREE.Raycaster` 等 → 改项目修改版 |
| 存在 setTimeout/异步过多 | 游戏逻辑应替换为 deltaTime/rAF 定时; 减少 async 调用栈深度 |
| 用防御式编程代替 fallback | 参数尽量必传，不满足条件尽早 throw，不用 `||`/`null` 兜底 |
| 减少可选参数 | 可选参数越多，调用方越容易遗漏，内部处理分支越多。考虑用必传 + 默认值工厂代替 |
| **状态同步** | 见下方子规则 |

##### 状态同步子规则

| 子规则 | 检测点 | 优先级 |
|---|---|---|
| 客户端擅自修改权威状态 | server 回包前就改了本地 hp/position/score | 🟡 |
| 发送增量而非绝对值 | 命令/协议字段是变化量（delta/offset）而非最终值 | 🟡 |
| 服务端未校验输入 | `executeCommand` 无速度上限/冷却/移动范围检查 | 🟡 |
| 断线状态未保留 | 玩家断开立即清 state，重连无法恢复 | 🟢 |

#### 🟢 关注项（只记不改）

| 规则 | 说明 |
|---|---|
| 条件嵌套 >3 层 | 加 `// TODO: 降低复杂度` |
| 导出函数缺 JSDoc | 补占位注释 |
| 空 catch 块静默吞错 | 记摘要 |
| 未清理的 event listener/timer/全局对象 | `window.__xxx` 挂载、未清理的 listener/timer，记摘要（内存泄漏风险） |
| 异步调用无超时/错误处理 | fetch/API 调用没有 timeout 或 catch，记摘要 |

输出格式：
```
## 代码审核 — Step 2：审查结果 + 重构方案

### 🔴 清理项（<N> 处）
- <文件>：<问题>

### 🟡 待确认重构（<N> 处）
- <文件>：<问题> → <方案>

### 🟢 关注项（<N> 处）
- <文件>：<问题>

**请兄弟确认：哪些 🟡 要改？**

⚠️ 兄弟对 🟡 项的回复规则：
- **没写内容** = 该项直接要重构
- **写了内容** = 按写的内容修改
- 只有写「不处理」「跳过」「忽略」等明确词才不处理
```

> ⏸️ **在此停住等确认。** 📣 发飞书通知（审核进度 + 🟡 重构项清单 + 请求确认）。
> 兄弟确认后（注意上方的回复规则），执行已确认的 🔴 + 🟡，完成后进入 Step 3。

### Step 3：BDD 测试

- 跑所有改动包的 BDD 测试（按改动范围选包，不跑无关包）：
  - `packages/frontend/` 有改动 → `npx jest --config jest.multiplayer.json`
  - `packages/room-service/` 有改动 → `npx jest --config jest.json`
  - `packages/match-service/` 有改动 → `npx jest --config jest.json`
- 测试不通过 → 修到全绿

### Step 3.5：编译检查

- 所有改动包分别跑 `npx tsc --noEmit` 检查类型编译错误
- 有错误 → 修到零错误，回头再跑 BDD 确认
- ⚠️ jest（ts-jest）只做转译不做类型检查，类型错误必须靠 `tsc` 暴露

### Step 4：输出审核摘要 + 提交 + 更新文档

输出格式：
```
## 代码审核摘要 — <日期>

**改动文件：** <N> 个
**🔴 已清理：** <N> 处
**🟡 已重构：** <N> 处（列出改动）
**🟢 关注项：** <N> 处
**测试结果：** ✅ BDD 全绿 / ✅ tsc 零错误 / ❌ <N> 失败
**重构效果：** <分析>

**请兄弟确认后更新文档**
```

> ⏸️ **在此停住等确认。** 📣 发飞书通知（审核摘要 + 请求确认更新文档）。
> 兄弟确认后：
> 1. **自审笔记产出** — 确认审核摘要/ADR/笔记文档内容准确、完整，修正后提交
> 2. 更新 `.last-review = HEAD`
> 3. `git add -A` + `git commit`（提交git，不 push）
> 4. 发飞书通知总结
> 5. 提示兄弟：「审核完成 ✅ 建议跑 e2e测试 确认联机环境无误，确认后推送」

---

## 执行纪律

1. **纯手动模式**——每步都停住等兄弟确认，不自动往下走
2. 🔴 清理项在 Step 2 方案中列出，兄弟确认后一起改
3. 🟡 重构必须先出方案，兄弟明确说改哪个才改（注意回复规则：没写内容=直接改，写内容=按内容改）
4. 🟢 只记录不改，不擅自动业务逻辑
5. 重构后测试不过 → 修到过，不能留烂摊子
6. Step 2 确认后 → 改代码 → Step 3 BDD（相关包） → **Step 3.5 tsc 编译检查** → Step 4 摘要 → **自审笔记** → 提交，顺序不能乱
7. 审核只审已有代码，不混入新功能
8. **保存协议不触发此 Skill**，代码审核是独立流程
