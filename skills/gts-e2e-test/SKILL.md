# E2E 手动测试（硬性操作规程）

> 触发词：兄弟说「e2e测试」/「e2e」/「e2e test」，可附带操作描述和验证目标。
> 双窗口手动操作，可选注入专用日志，点停止后抓数据验证。

---

## 步骤

### Step 0：重启服务（必须）

**每次运行 E2E 前必须先重启 room-service + match-service：**
1. Kill 现有 room (4003) + match (3000) 进程
2. 启动 room-service (`yarn dev`)，等启动完成
3. 启动 match-service (`yarn dev`)，等连接成功

不重启会导致 match-service WS 失连，游戏卡在"查找房间中"。

### Step 1：解析输入（可选）

兄弟可选择附带验收标准，格式同 e2e自动：

```
e2e测试：
操作巨大娘往小人方向移动碰撞；
验证：小人血量至少减少5点
```

**有验收标准时：** 设计专用日志 hook（同 gts-e2e-auto Step 1），采集验证所需数据。**无验收标准时：** 跳过日志设计，走纯手动模式。

### Step 2：生成并启动测试脚本

**有验收标准：** 以 `e2e-scene2-manual.cjs` 为模板，注入 Step 1 设计的专用日志 hook + 停止时数据采集逻辑。生成新脚本保存到 `test/e2e/manual/`。

**无验收标准：** 直接跑原始 `e2e-scene2-manual.cjs`。

以 background 模式运行，启动后告知兄弟。

### Step 3：等待测试结束

- 不 poll，等脚本自然退出
- 用户手动操作，点停止按钮结束

### Step 4：验证 + 会话分析

**有验收标准：** 脚本退出后读取采集数据，对照验收标准输出分析。**无验收标准：** 直接发飞书通知。

---

## 执行纪律

1. **运行前必须先重启 room + match**（Step 0 强制执行）
2. 无验收标准 → 跑原始脚本，不设计日志
3. 有验收标准 → 注入专用 log hook，动态生成脚本并保留
4. 等待期间**不 poll、不催促**
5. 结果分析**在会话中给出**，飞书只发短提醒
6. 和 `gts-e2e-auto`（全自动版）互不触发
