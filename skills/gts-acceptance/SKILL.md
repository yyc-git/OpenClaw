# 验收流程（硬性操作规程）

> 触发词：兄弟说「验收：<标准>」或「验收」。
> 自动循环模式：改 → 测+验证标准 → 修，直到通过或超时。

---

## 步骤

### Step 1：接收验收标准

- 兄弟发送验收指令时，附带通过标准（如「所有 BDD 全绿」「某场景无报错」「血条不闪烁」等）
- 默认通过标准：**全部 BDD 测试通过**（`npx jest --config jest.multiplayer.json`）
- 默认总超时：**1 小时**

### Step 2：进入改-测-验循环

```
┌─ 分析失败原因，修改代码 ──────┐
│                                ↓
│  1. 运行全部 BDD 测试           │
│  2. 逐条验证兄弟指定的验收标准    │
│                                ↓
│  测试全绿 + 标准全部达标？──→ ✅ 进入 Step 3
│  任一未达标                    ↑
│  超时？──→ ❌ 进入 Step 3（标记超时）
└──否──→ 继续循环 ───────────────┘
```

- 每次修改后：**先跑测试，再逐条对照验收标准**
- BDD 全绿但标准未达标 → 不算过，继续修
- 标准达标但 BDD 挂了 → 不算过，继续修
- 两者都过 → 跳出循环
- 超过 1 小时 → 跳出循环，标记超时

### Step 3：输出结果 + 飞书通知

| 结果 | 飞书内容 |
|---|---|
| ✅ 通过 | `验收通过 — BDD 全绿 + 验收标准全部达标` |
| ⏰ 超时 | `验收超时（1h）— <N> 次尝试，BDD: <M/N> 通过，标准: <X/Y> 达标，当前阻塞: <简述>` |

---

## 执行纪律

1. **没有「跳过」**——验收循环必须跑到通过或超时
2. **每次修改后强制两件事**：跑测试 **+** 逐条验证验收标准，缺一不可
3. 测试全绿但标准未达标 ≠ 通过
4. 超时后不等确认，直接通知兄弟
5. 验收通过后立即飞书通知
6. 验收过程中不混入新功能
