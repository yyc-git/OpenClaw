## WebGPU 与多线程迁移全分析

> 兄弟逐轮追问 9 轮，全面分析了 GTS-Play 多人联网从 WebGL 迁 WebGPU + 多线程的可行性

### 关键结论

**WebGPU 迁移**：
- Three.js 0.184 成熟度 ~75%，多人路径够用
- `WebGPURenderer` 自带 WebGL2 自动回退
- 多人路径迁移改动 ~15 行（`ThreeRenderer.ts`），工期 2-3 天
- 移动端 iOS Safari 26.0+ 终于支持 WebGPU，但 QQ/UC/百度浏览器不支持（走回退）

**GPU-Driven Pipeline**：
- Three.js 0.184 积木齐全：ComputeNode/StorageBuffer/IndirectDraw/Atomic
- GPU 碰撞检测 把握 75% ← 推荐第一优先级
- GPU LOD 把握 85%
- Instance Frustum Cull 把握 75%
- Hi-Z 遮挡剔除 把握 55%
- Meshlet Triangle Cull 把握 25%（研究级）

**多线程**：
- Logic Worker (OBB/寻路) 把握 90%，现在就能做
- Data-Oriented SAB + OffscreenCanvas 架构可行（70%），但工程量大
- SAB ~450KB，双缓冲 + AtomicNotify 零锁同步

**迁移铁律（现在就该做的）**：
1. 动画用纯数据接口 ✅ 已做到
2. Entity 状态 TypedArray 化
3. 新功能走 TSL，不走 GLSL
4. 不走 WebGL 专用 API
5. 不碰 WebGLRenderTarget
6. 渲染层不持有游戏状态 ✅ 已做到
7. Game Logic 不直接碰 Three.js ✅ 已做到

**成本公式**：
- 现在克制 = 将来 3-4 周
- 现在复刻单机功能 = 将来 9-14 周（每加一个 WebGL 专用 API 将来要还 2-3 倍）

## WebGL E2E 调试体系建设

> 兄弟要求实现 E2E 测试中直接监控 WebGL draw call、shader 等 GPU 信息
> 参考：BrowserAct、RenderDoc、renderdoc-mcp、Spector.js、虚幻引擎 AI D3D Debug

### BrowserAct 评估

- **不是 Playwright 平替**，是 AI Agent 专用浏览器 CLI + 反检测基础设施
- 三层反检测：指纹伪装/TLS轮换 → solve-captcha → remote-assist人工接管
- 三种浏览器模式：Chrome（复用登录态）、Stealth Privacy（无痕）、Stealth Fixed Identity（多账号隔离）
- Token 优化：索引化文本替代完整 HTML
- **GTS-Play 决策：E2E 继续用 Playwright，不迁移**
- 分析存入：`笔记/决策记录/BrowserAct评估-2026-06-20.md`

### WebGL 调试体系实施（四层）

**Layer 1 — 聚合统计** ✅
- `ThreeRenderer.getGLStats()` — 从 `renderer.info` 导出 drawCalls/triangles/textures/programs
- `window.__GL_STATS__` 每帧刷新
- E2E: `page.evaluate(() => window.__GL_STATS__)`

**Layer 2 — Shader 编译错误捕获** ✅
- `WebGLDebug.installShaderErrorHook()` — 包装 `compileShader`
- GLSL 编译错误自动写入 `window.__SHADER_ERRORS__`
- Playwright 注入版: `test/helpers/gl-shader-hook.cjs` (addInitScript)

**Layer 3 — Draw Call 按类型分桶** ✅
- `scene.traverseVisible` → Mesh/Line/Sprite/SkinnedMesh 分类
- 实战：22 DC = 18 Line (Grid+OBB线框) + 2 SkinnedMesh (角色) + 2 MeshBasic (碰撞盒)
- 关掉 debug 后预计 ~4 DC

**Layer 4 — Shader 源码提取** ✅
- 从 `renderer.info.programs` 拿 WebGLProgram
- `gl.getAttachedShaders` + `gl.getShaderSource` 提取 GLSL
- 附带 `usedTimes`，一眼看出哪个 shader 最忙
- 实战：8 个 programs，主力 MMD 蒙皮 shader 用 7x

### Spector.js 尝试（失败）

- WebGL2 兼容问题：`onCapture` 回调始终不触发
- `page.route` HTML 注入 + `addInitScript` 均无法可靠拦截
- 根因：游戏 WebGL 上下文创建时机早于注入脚本生效
- 660KB 体积 + 30s 页面加载延迟 = 不适合 E2E

### GLTracer 自研工具（代码完成，被 webpack 缓存挡住）

- 嵌入 ThreeRenderer 初始化流程，不依赖 Playwright 注入
- TypeScript 源码：`scene3d_layer/utils/GLTracer.ts`
- 功能：自动包装 drawArrays/drawElements/useProgram/bindTexture 等
- `window.__GL_TRACE__.captureMs(3000)` → 返回逐条 GL 命令
- **当前状态**：webpack-dev-server 的 `transpileOnly`+`happyPackMode` 缓存不识别新文件
- 解决方法：清 webpack 缓存或改配置，待兄弟决定

### 关键教训

1. **WebGL 上下文时序死锁**：Playwright `addInitScript` 理论在页面脚本之前执行，但 React/webpack 应用的模块初始化可能在 DOM ready 之前就创建了 WebGL 上下文
2. **Spector.js 不适合 E2E**：666KB 体积、30s 加载延迟、WebGL2 兼容问题
3. **直接挖 Three.js 内部数据 > 拦截 GL 命令**：`renderer.info` + `scene.traverseVisible` + `gl.getShaderSource` 已覆盖 95% 场景
4. **webpack-dev-server 缓存**：新文件不被识别，`transpileOnly`+`happyPackMode` 下需要清缓存
